# UUIDシークレットキー方式のアカウント管理

メール・パスワード不要の低摩擦アカウント。UUID＝ログインキー＝アカウントそのもの。
小〜中規模の参加型Webツール（イベント運営・アンケート等）に向く。

## サーバー側の原則

- キーは `crypto.randomUUID()` で生成。**サーバーはSHA-256ハッシュのみ保存**（生の鍵は発行レスポンスでしか得られない）。`key_hash` にUNIQUEインデックス
  - **SHA-256（ソルト無し・高速）で足りるのは、キーが 122bit の高エントロピー乱数だから**（総当たり・レインボーテーブルとも無効）。**この理屈をユーザーが決めるパスワードに一般化しないこと**。パスワードは低エントロピーなので bcrypt/argon2 等の遅いハッシュ＋ソルトが必須
- 認証はミドルウェアで一元化: ハッシュ照合 → 不一致は401。認証済みユーザーは `ctx.data` に格納
  - **キーは必ず HTTP ヘッダで受ける（例 `x-org-key`）。`?key=` クエリで受けない**。クエリ文字列はページ/APIリクエストでエッジに届き、**サーバ/CDN のアクセスログにキー（＝ベアラー）がそのまま残る**（ログ流出・閲覧権限者経由で乗っ取り）。`history.replaceState` で URL から消してもログ記録は防げない
  - **別端末用の専用ログインリンクは `#key=`（フラグメント）で**。フラグメントはサーバに送られないのでログに残らない。読み込み時に JS で `location.hash` を読んで localStorage 保存 → hash を消す。`?key=` は既配布リンクの後方互換フォールバックとしてのみ受ける（新規は発行しない）
- 所有権チェックを全エンドポイントで: 他人のリソースは**404**を返す（403だと存在が漏れる）
- キーをログに出さない。URLにキーが乗る設計なら `Referrer-Policy: no-referrer` を必ず付ける

## クライアント側の原則

- キーは localStorage（オリジン単位であることに注意。**プレビューデプロイURLは別オリジン**＝別ストレージ）
- 専用リンクログインは `#key=`（フラグメント）: 読み込み時に `location.hash` から読んで localStorage へ保存し、**即 hash を消す**（`history.replaceState`）。`?key=` は既配布リンクの後方互換フォールバックのみ（読み取り時に併せて受ける／新規発行はしない）
- ログアウト＝localStorageから削除。「キーを保存していないと二度と入れない」の強い確認ダイアログ必須
- キー管理パネル: マスク表示＋表示切替・コピー・専用リンクコピー・メール保存（Gmail compose URL）・`navigator.share`
- Safari は約7日間未訪問で localStorage を消すことがある（ITP）。「キーを別の場所にも保存して」の案内が保険になる

## パスワードマネージャ対応（Chrome内蔵・iCloudキーチェーン・1Password共通）

**前提：PM は「対応する」が「主役にしない」。** キー保存の主役は**手動（コピー／メール／LINE／`navigator.share`）**にする。理由は2つ：(1) PM を使っていない利用者もそれなりにいて、PM前提だとその人たちには無意味、(2) PM 利用者でも Safari(ITP)・シークレット・ブラウザ差で**保存提案が出ないことがある**。だから PM も効くように**必要な項目は HTML に入れておく**（下記の form 化・`autocomplete`・キー保存ステップ）が、それを頼りきりの導線にはしない。「PM が拾えたら儲けもの、拾えなくても手動で確実に保存できる」の二段構えにする（実地の結論）。以下は「PM も効かせる」ための具体作法。

- 保存トリガーは「password欄を含む `<form>` の **submit**」。裸のinput＋button onclickでは不安定
- submitだけでは不十分: PMは「submit後のログイン成功」を検知して初めて保存提案する。
  最強のシグナルは**実ナビゲーション**。SPAのdiv差し替えだけだと提案が出ないことがある
  （同一オリジンに同じパスワードの既存エントリがあると特に黙殺されやすい）。
  ログイン成功後に `location.reload()` か実遷移を入れるのが確実（認証情報がstorageにあれば
  リロード後はそのままログイン後画面を出せる）
- ログインフォーム: ユーザー名欄（`autocomplete="username"`）＋キー欄に `autocomplete="current-password"`
- **PMは可視性ヒューリスティクスで欄を選ぶ**: `type=hidden`/`display:none` はほぼ全PMが無視。
  画面外（`left:-9999px`）はChrome内蔵は概ね可だが Safari・1Password・Bitwarden 等はスキップすることがあり、
  `aria-hidden="true"` も「実在しない欄」とみなすPMがある。
  **固定ユーザー名は隠さず、見える readonly 欄にするのが最も確実**（薄いグレー表示などで控えめにすればよい）
- 保存済みエントリは保存時のスナップショット: ユーザー名欄が不可視のうちに保存されたものは
  「ユーザー名が指定されていません」のまま残り、フォームを直しても自動では直らない。
  可視化後に再ログインすると更新提案が出る（またはPM側でエントリを手動編集）
- **キー保存ステップ**（発行直後に1回だけ挟む画面）: username=表示名 readonly ＋ password=キー readonly `autocomplete="new-password"` のformを見せ、submitボタン（preventDefault→遷移）を押させる。readonlyでも保存される
  - **登録時は必ずこのフェーズを通す**。登録完了→直接アプリへ遷移はNG（保存提案の機会が一度も無いままキーが消えると復旧不能）。遷移は必ずこの画面のsubmit経由にする
  - ブラウザが保存を提案しないケースもある（設定・拡張・ブラウザ差）。「提案が出ない場合は次ページのキー管理パネル（コピー・専用リンク・メール等）で保存できる」旨を画面内に併記する
- `navigator.credentials.store()` はChrome系のみ（Safari・1Password非対応）。formと併用すると二重提案になるので基本はform一本
- オートフィルされたくない欄（例: 別キーの入力欄）には `autocomplete="off"`
- IME注意: テキスト欄の keydown Enter は変換確定で誤発火する。form submit 経由なら安全

### 単一トークン認証（ADMIN_TOKEN等）もパスワードマネージャ対応にする

- 「ユーザー名なしの共有トークン」型の認証（管理者の ADMIN_TOKEN 等）も、
  ログインを form 化すればパスワードマネージャで扱える:
  見える readonly ユーザー名欄（固定値・下記の衝突回避命名）＋ トークン欄に `autocomplete="current-password"` ＋ type=submit
- トークン自体の保持は sessionStorage（タブを閉じると消える）のままでよい。
  「マネージャがトークンを記憶・オートフィル」×「セッション中だけ保持」の組み合わせが
  再入力の手間とセキュリティのバランスが良い

### ユーザー名の衝突に注意（エントリは「オリジン＋ユーザー名」単位）

- パスワードマネージャは同一オリジン・同一ユーザー名を**同じエントリ**とみなし上書き提案する
- 管理者ログイン等の固定ユーザー名に `admin` のような**利用者が表示名として入力しうる値を使わない**。
  `<サービス名>-site-admin` のようにプレフィックスを付ける
  （例: 利用者が表示名「admin」で登録→同じブラウザで管理者ログイン→ADMIN_TOKENのエントリと衝突・上書き）
- 一般ユーザーのログインフォームの不可視ユーザー名は固定値（例「主催者」）でよいが、
  同一ブラウザで複数アカウントを使い分けるとログイン時保存がその名義で上書きされ合う。
  発行直後の「キー保存ステップ」は実際の表示名で保存するので、そちらが各アカウントの本命エントリになる

## リカバリ設計

- 自己サービスの再発行は無し（本人確認手段が無いため原理的に不可能）
- **管理者経由の再発行**: 新UUID発行→ハッシュ置換（旧キー即無効）→ログインURL（`#key=`付き）を管理画面に1回だけ表示し、管理者が本人に渡す
- **「再発行」は2種類を区別する**:
  - **ロスト回復**（鍵を無くして締め出された）＝本人確認手段が無いので**自己サービスでは不可能**。上の管理者経由のみ
  - **ローテーション**（ログイン中の本人が、漏れたかもしれない鍵を自分で差し替える）＝本人性は担保されているので**可能**。新UUID発行→ハッシュ置換→旧キー即無効→各端末で新キーに入れ直す。**注意**: 鍵が既に漏れていると攻撃者も同じく「本人」としてローテーションでき、先にやられると締め出される（レース）。低機微前提なら許容
- アカウント統合（マージ）は必要になってから。実装した例もあるが、多くの用途では不要

- [ ] 認証ミドルウェア＋全エンドポイントの所有権チェック（他人は404）
- [ ] 専用リンクは `#key=`（フラグメント）＝ログに残らない。読込時に localStorage 保存後 hash を即消去
- [ ] ログアウトの強い確認
- [ ] ログインフォームのform化＋autocomplete属性
- [ ] 発行直後のキー保存ステップ（form submit経由・**登録フローで必須経由**・保存提案が出ない場合の代替手段を併記）
