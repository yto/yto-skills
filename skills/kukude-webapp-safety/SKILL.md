# 公開Webアプリの致命傷を避ける（初心者向け）

個人・小規模で**公開する**Webサイト/アプリを作る人を、Claude が実害から守るためのガイド。網羅的なセキュリティ監査ではなく、**「これだけは踏んではいけない地雷」を避ける最低限**。

## スタンス

- **過剰に縛らない。止めるのは「取り返しのつかない失敗」だけ。** 便利さや手軽さは活かしてよく、小さな失敗はやり直せる。安全を理由に何もさせないのではなく、後戻りできない事故だけを未然に防ぐ。
- **致命傷 = 取り返しがつかない/被害が大きいもの**：秘密（APIキー・トークン・パスワード）の漏洩、個人情報の流出、他人による不正利用・課金事故。
- ⚠️ **これは最低限であって、これだけで安全が保証されるわけではない。** 決済・健康・他人の個人情報など**重い情報を扱うなら、作る前に詳しい人に相談する**よう促す。

## Claude としての振る舞い

- 危険になりうる操作（秘密をコードに書く、リポを Public にする、認証を自前で作る等）の**前にリスクを先に指摘**し、安全な代替を提案する。
- 非エンジニアにも**平易に**説明する。専門用語は避けるか、ひとこと補足する。
- 下の原則に反する実装をユーザーが望んでも、まず理由を説明し、代替案を出してから進める。

## 避けるべき致命傷（原則）

1. **秘密のキー・トークンをソースコードに直書きしない。**
   - なぜ：漏れると不正利用・課金事故に直結する。Public リポなら即バレる。一度コミットすると Git 履歴に残り、消すのは難しい。
   - どうする：秘密は**サーバ側の環境変数/Secrets** に置く（例：Cloudflare の Secrets、`.env` は `.gitignore`）。フロントのJSに秘密を置かない（ブラウザに送ったものは誰でも見られる）。

2. **個人情報をリポジトリにコミットしない。**
   - なぜ：Public なら流出。Private でも履歴に残る。メールアドレス・氏名・電話番号・顧客名簿など。
   - どうする：個人情報はコード/リポに入れず、必要ならDBに。テストデータにも実在の個人情報を使わない。

3. **ログインの合否をブラウザ側（JavaScript）だけで判定しない。**
   - なぜ：ブラウザに送ったコードやデータは誰でも書き換え・閲覧できる。JSの`if`で「ログイン成功」にしても素通りされる。
   - どうする：**認証・認可の判定は必ずサーバ側**（API/Functions）で行う。フロントの制御は見た目だけと考える。

4. **リポジトリは基本 Private にする。**
   - なぜ：書きかけのコードや、うっかり入れた秘密・個人情報の被害を限定できる。
   - どうする：既定は Private。**GitHub Pages で公開する等、仕組み上 Public が必須のときだけ** Public にし、そのリポには秘密・個人情報を絶対に入れない。

5. **ユーザーが決めたパスワードを自前で保存する認証は避ける。**
   - なぜ：漏洩リスクが高く、安全に作り切るのは難しい（ハッシュ・ソルト・リセット・漏洩対応…）。
   - どうする：**パスキー**（＋リカバリコード）や、**ランダムなログインキー（UUID）方式**（メール不要・パスワード不要）を使う。ただしログインキー方式は**漏れても致命的でない情報のアカウントに限る**。

## 認証は「そもそも要るか」を先に分ける

作り始める前に切り分ける。無駄な認証は作らない＝作り込みの地雷を減らす。

- **多くの公開アプリは認証不要**（誰でも見られる/使えるほうがよい宣伝サイト・公開ツールなど）。
- 認証が要るのは、**自分だけが使う管理画面**や、**ログインした本人だけが見る個別ユーザー画面**など、相手を絞りたい部分だけ。
- 認証が必要になったら、上の原則5の方式（パスキー / UUIDキー）を検討する。管理画面を数人で守るだけなら、プラットフォームの認証（例：アクセス制御サービス）に任せる手もある。

## 使わない範囲（このスキルの対象外）

会社の業務システム・社内向けアプリ・商用グレードのシステムは対象外。その用途は、所属先のルールや情報システム部門・セキュリティの専門家の方針に従う。ここで扱うのは**個人・小規模で公開する**場面に限る。
